# Project Documentation System Design

## 1. 背景与目标

当前仓库已有 PRD、架构蓝图、前后端说明文档，但信息分散在根目录、`docs/`、`backend/`、`kitchen-tablet/`。目标是建立一套可持续维护的文档体系，让新成员可以按固定路径快速理解项目并完成开发/部署/联调。

本次范围是“项目级 + 模块级”文档完善：新增项目核心专题文档，补齐模块入口文档，并统一导航与职责边界。

## 2. 设计原则

1. 单一事实源：同一主题只在一个文档维护，其他文档只做引用。
2. 入口分层：根 README 给全局入口，`docs/README.md` 给专题入口，模块 README 给操作入口。
3. 可执行性：开发、部署、测试文档包含前置条件、命令、结果和常见失败处理。
4. 与现状一致：API 以当前 FastAPI 路由为准，不引入未实现能力。
5. 最小重复：PRD/ADR 保持既有定位，不被“操作手册类文档”覆盖。

## 3. 文档架构

### 3.1 入口层

- `README.md`（根）：仓库总览、阅读顺序、快速跳转
- `docs/README.md`：文档中台导航、文档职责与更新约定

### 3.2 项目专题层（集中在 `docs/`）

- `docs/architecture-overview.md`：系统上下文、组件边界、数据流
- `docs/api-reference.md`：REST/WS 契约、鉴权、错误语义
- `docs/development-guide.md`：本地开发、联调、目录约定
- `docs/deployment-runbook.md`：部署流程、环境变量、回滚
- `docs/testing-strategy.md`：测试分层、执行命令、准入标准
- `docs/troubleshooting.md`：常见故障、排查路径、恢复动作

### 3.3 模块层

- `backend/README.md`：后端启动、测试、部署入口与跳转
- `kitchen-tablet/README.md`：前端启动、构建、联调入口与跳转

## 4. 内容模板标准

每份专题文档统一结构：

1. 适用范围（谁用、什么时候用）
2. 前置条件（环境、依赖、权限）
3. 核心流程（步骤/接口/约束）
4. 错误与异常处理（失败场景与动作）
5. 关联文档（上游/下游链接）

这保证读者从任意入口都能快速判断“是否该读这篇”和“下一步去哪里”。

## 5. 一致性规则

1. API 路径必须与 `backend/app/api/*.py` 一致。
2. 运行命令优先来自各模块现有脚本/README，避免发明新命令。
3. 文档术语统一使用：订单状态流转、催单、快照、WebSocket 广播。
4. 任何部署路径说明必须标注可配置项（如 `DB_PATH`）。
5. 不在文档中承诺尚未落地的功能。

## 6. 验收标准

1. 新成员可按“根 README → docs/README → 专题文档”完成端到端理解。
2. 至少覆盖：架构、API、开发、部署、测试、排障六类专题。
3. `backend/README.md` 与 `kitchen-tablet/README.md` 保持模块操作导向。
4. 文档链接可达、文件名一致、无明显重复段落。

## 7. 非目标

1. 不改动业务代码和接口行为。
2. 不重写 PRD、ADR、技术蓝图原文。
3. 不引入流程平台化工具（如文档站点构建器）。

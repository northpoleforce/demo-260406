# 产品需求文档（PRD）

## 0. 文档信息
- 产品名称：边缘 AI 智能餐饮交互系统（MVP）
- 文档版本：v1.1（按“云端 ASR + 餐桌仅语音催单”更新）
- 文档状态：评审中
- 目标上线形态：单店局域网离线可运行
- 目标上线窗口：T0 + 6 周（含试点）

## 1. 业务背景与目标
### 1.1 业务痛点
- 顾客无法实时感知菜品进度，催单频繁，影响体验与服务效率。
- 后厨状态流转不透明，人工口头传达导致延迟和错漏。
- 断网场景下 SaaS 系统不可用，营业连续性风险高。

### 1.2 MVP 目标（6 周内达成）
- 订单状态在餐厨端与桌牌端实时同步，P95 延迟 <= 300ms，P99 <= 500ms。
- 催单闭环（仅语音催单）可用，云端 ASR 意图识别准确率 >= 88%（店内噪声条件下）。
- 顾客问询量下降 >= 30%，后厨无纸化状态流转覆盖 >= 90%。

### 1.3 非目标（MVP 不做）
- 不接入外卖平台和收银系统深度对接。
- 不做云端多门店统一调度。
- 不做视觉出餐识别（放到下一阶段）。

## 2. 用户角色与核心场景
### 2.1 用户角色
- 顾客：关注本桌菜品进度，发起催单。
- 厨师/配菜员：处理订单，更新状态。
- 店长/管理员：监控系统在线状态，处理异常与导出日志。

### 2.2 关键业务场景（MVP）
1. 顾客下单后，在桌牌端看到状态变化：排队中 -> 制作中 -> 即将上桌 -> 已上桌。
2. 顾客语音催单后，后厨看板高亮并有声光提醒。
3. 后厨点击状态流转后，所有在线终端在 500ms 内收到更新。
4. 平板短时离线后自动重连并拉取最新快照，状态不丢失。
5. 断电重启后 60 秒内恢复服务，订单状态与断电前一致。

## 3. 范围边界与软硬件职责
### 3.1 系统边界
- 本系统负责：订单状态管理、催单闭环、多端同步、离线可运行。
- 外部系统负责：点单交易、支付、会员、库存。

### 3.2 软硬件边界（必须明确）
- 边缘主控（地瓜开发板）：
    - 承载 API、状态机、WebSocket、SQLite、云 ASR 适配与结果回传服务。
    - 不承担 UI 渲染。
- 餐厨端（平板网页）：
    - 仅做展示与交互，不存储核心业务状态。
- 桌牌端（网页/微屏）：
    - 展示本桌状态，采集语音催单输入。
- 语音节点（麦克风阵列/USB 麦）：
    - 采集音频并流式发送，不做本地推理。

### 3.3 失效兜底边界
- 云 ASR 不可用时，餐桌端提示“语音服务暂不可用，请联系服务员”，不得影响基础状态流转。
- WebSocket 不可用时，前端降级为 2 秒轮询。

## 4. 总体架构
### 4.1 逻辑组件
1. API 网关（FastAPI + Nginx）
2. 状态机服务（订单生命周期与幂等校验）
3. 实时通道（WebSocket 广播 + 心跳）
4. 存储层（SQLite WAL + 定时快照）
5. 语音服务（音频接入 + 云 ASR API 适配 + 意图分类）
6. 运维监控（健康检查、日志、告警）

### 4.2 订单状态机定义
- 状态集合：`queued`、`cooking`、`ready`、`served`、`urge_requested`。
- 合法迁移：
    - `queued -> cooking -> ready -> served`
    - 任意非 `served` 状态可附加 `urge_requested` 标记（而非替换主状态）。
- 非法迁移处理：返回 409，并记录审计日志。

### 4.3 关键数据模型（SQLite）
- `orders(order_id, table_id, created_at, current_state, version)`
- `order_items(item_id, order_id, dish_name, qty, item_state)`
- `urge_events(event_id, order_id, table_id, source, confidence, created_at)`
- `device_status(device_id, device_type, last_seen_at, is_online)`
- `audit_logs(log_id, actor, action, payload, created_at)`
- `asr_calls(call_id, table_id, provider, latency_ms, success, error_code, created_at)`

## 5. AI 推理与算力分配（可落地方案）
### 5.1 原则
- 业务连续性优先于模型精度。
- 云端推理与本地业务线程解耦，避免语音链路阻塞主链路。
- 可观测优先，确保可跟踪云接口可用性和成本。

### 5.2 推理策略（MVP 固定方案）
- 采用云端 ASR API 进行识别，本地仅做音频采集、传输、结果落库与业务触发。
- 云端返回文本后，本地执行轻量意图分类（仅“催单”及否定/无效语义过滤）。
- 云端调用超时控制：单次请求 1200ms 超时，最多重试 1 次（指数退避）。

### 5.3 资源配额（地瓜开发板）
- CPU：
    - Core 0-1：API + WebSocket + DB
    - Core 2-3：音频接入、编码、云接口调用与结果处理
- BPU/NPU：MVP 阶段不启用（预留后续本地推理扩展）
- 内存预算：
    - 系统与服务基础占用 <= 45%
    - 语音调用峰值后总内存占用 <= 75%
- 保护机制：
    - 当云 ASR 错误率 > 30% 持续 5 分钟，触发语音服务降级告警并提示人工服务。

## 6. 详细功能需求
### 6.1 边缘主控服务
- 局域网组网：支持静态 IP 与 mDNS 发现。
- 对外接口：REST + WebSocket。
- 心跳机制：客户端每 10s 心跳，丢失 3 次判定离线。
- 重连策略：指数退避（1s/2s/4s，最大 10s）。
- 数据一致性：状态更新采用乐观锁 `version` 防并发覆盖。

### 6.2 餐厨端（平板 Web）
- 看板排序：默认按下单时间，可切换“催单优先”。
- 一键流转：按钮大于 48px 点击区，二次确认可配置。
- 催单提示：红色边框 + 蜂鸣 1.5 秒，可静音 5 分钟。
- 离线标识：明显水印，恢复后自动全量同步。

### 6.3 桌牌端（Web/微屏）
- 仅展示本桌数据，不允许查看其他桌。
- 催单交互：
    - 仅语音催单：显示“识别中/已通知后厨/识别失败请重试/语音服务暂不可用”。
- 防误触：30 秒内重复催单仅保留一次有效上报。

### 6.4 语音交互节点
- 输入：16kHz 单声道 PCM。
- 前处理：VAD 截断 + 可选降噪，压缩后发送至边缘主控。
- 判定：云 ASR 返回置信度阈值默认 0.75，可后台配置。
- 错误文案：低置信度提示“环境嘈杂，请靠近麦克风重试或联系服务员”。

## 7. 接口契约（MVP）
### 7.1 REST
- `POST /api/v1/orders/{id}/state`
    - 入参：`target_state`, `operator`, `version`
    - 出参：最新订单快照
- `POST /api/v1/orders/{id}/urge`
    - 入参：`source(voice)`, `confidence?`, `asr_provider`, `asr_latency_ms`
    - 出参：`accepted: true|false`, `reason`
- `POST /api/v1/voice/asr/callback`
    - 入参：`request_id`, `table_id`, `text`, `confidence`, `provider`
    - 出参：`received: true|false`
- `GET /api/v1/snapshot?since=timestamp`
    - 出参：增量或全量状态

### 7.2 WebSocket 事件
- 频道：`/ws/store/{store_id}`
- 事件：`order.updated`、`urge.created`、`device.offline`、`system.degraded`
- 可靠性：客户端按 `event_id` 去重，服务端支持最近 2000 条事件重放。

## 8. 稳定性、可运维性与安全
### 8.1 稳定性 SLO
- 系统可用性（营业时段）：>= 99.5%
- 断电恢复时间 RTO：<= 60s
- 可容忍数据丢失 RPO：<= 1s（WAL + flush 策略）

### 8.2 监控与告警
- 指标：CPU、内存、温度、WS 连接数、消息堆积、云 ASR 调用延迟、云 ASR 成功率、ASR 置信度分布、ASR 调用成本。
- 告警阈值：
    - P95 同步延迟 > 500ms 持续 5 分钟
    - 离线设备数 > 20%
    - 云 ASR 成功率 < 90% 持续 10 分钟

### 8.3 安全最小集
- 局域网内设备鉴权：预共享 token + 设备 ID 白名单。
- 管理端操作审计：所有状态变更写入 `audit_logs`。
- 云接口安全：ASR API Key 存储于设备安全区，按月轮换。
- 日志脱敏：不记录原始语音，仅记录摘要、结果与错误码。

## 9. 异常与降级策略
- 高噪声：低置信度不触发催单，提示靠近麦克风重试或联系服务员。
- 云 ASR 超时（> 1.2s）：本次语音失败，提示用户重试或联系服务员。
- DB 锁冲突：重试 3 次，失败后返回可理解错误并引导刷新。
- 网络抖动：自动重连并拉取快照，确保最终一致。
- 外网中断：语音催单不可用，状态流转与看板同步继续可用。
- 设备过热：触发 `system.degraded`，限制语音并发，优先保证核心业务。

## 10. 性能指标与验收标准
### 10.1 SLA
- 触控状态变更 -> 多端可见：P95 <= 200ms，P99 <= 500ms。
- 语音结束 -> 后厨收到催单提醒：P95 <= 2.0s。

### 10.2 UAT 验收用例（必须通过）
1. 20 台终端同时在线，连续 30 分钟状态流转无错乱。
2. 外网断开情况下，状态流转可完整跑通 50 单；语音催单正确提示不可用。
3. 断电重启后 60 秒内恢复，订单状态与断电前一致。
4. 高噪声环境下误触发率 <= 5%。
5. 任一终端离线 10 分钟后重连，数据最终一致。

## 11. 实施计划（6 周）
### Week 1-2：基础链路
- 完成状态机、REST、WebSocket、SQLite WAL。
- 交付：可跑通无语音催单的全链路演示。

### Week 3-4：语音闭环
- 接入麦克风、VAD、云 ASR API、催单意图识别。
- 交付：店内试运行，语音可用率达标。

### Week 5：稳定性加固
- 完成离线重连、断电恢复、监控告警、压力测试。
- 交付：灰度报告与问题清单清零。

### Week 6：试点上线
- 单店营业时段运行，收集 KPI（问询量、催单响应时延）。
- 交付：上线复盘与云 ASR 成本/稳定性优化计划。

## 12. 风险与应对
- 风险：店内噪声导致 ASR 波动。
    - 应对：关键词+置信度双阈值，失败提示重试或联系服务员。
- 风险：硬件温度高导致降频。
    - 应对：温控告警、语音并发限流、风扇改造。
- 风险：并发更新导致状态覆盖。
    - 应对：乐观锁 version + 幂等 event_id。
- 风险：云 ASR 服务波动或配额不足。
    - 应对：双供应商预案、配额监控、熔断和故障告警。

## 13. 待确认项（请产品/业务本周确认）
1. 单店峰值并发终端数（当前按 20 台设计）。
2. 单店峰值订单吞吐（当前按 120 单/小时设计）。
3. 是否允许顾客连续催单（默认 30 秒节流）。
4. 云 ASR 供应商选择（单供应商/双供应商容灾）。
5. 云 ASR 费用预算上限（按日/按月）。
6. 试点门店噪声基线（分贝范围）与麦克风布点。

## 14. 结论与推荐决策
- 推荐路线：餐桌仅保留语音交互，ASR 全量走云端 API，本地专注业务链路与稳定性。
- 推荐原因：
    - 减少本地模型部署和调优成本，缩短首版交付时间。
    - ASR 精度和模型迭代由云服务托管，维护负担更低。
    - 本地资源集中保障状态同步与可用性，主链路更稳。
